# FraudMesh — Product Requirements Document
### Adaptive Real-Time Fraud Detection Using Graph-Based Behavioral Analytics
**Version:** 1.0 | **Author:** Daksh | **Track:** FinTech — Fraud Detection | **Status:** Hackathon Prototype

---

## 1. Executive Summary

FraudMesh is an adaptive, real-time fraud detection platform that models financial entities and their relationships as a living graph — detecting fraud not by looking at individual transactions in isolation, but by analyzing the behavioral network surrounding every transaction. When a fraud ring operates, it leaves a web of signals across accounts, devices, IPs, and merchants. FraudMesh is built to see that web.

The platform combines Graph Neural Networks for relational anomaly detection, a context-aware adaptive threshold engine, and Claude Opus 4.5 as an explainability layer that translates complex fraud patterns into plain-English analyst reports — all in under 200 milliseconds.

---

## 2. Problem Statement

Modern financial fraud is no longer a single-actor, single-transaction crime. It has evolved into coordinated network-based operations — fraud rings, synthetic identity clusters, money mule chains — that look completely normal when examined transaction by transaction. Traditional rule-based fraud systems and isolated ML models are fundamentally blind to these patterns.

At the same time, over-sensitive fraud systems cause false positives that block legitimate users, eroding trust and damaging customer experience. The challenge is detecting coordinated fraud early and accurately, while keeping false positive rates low and providing human-interpretable explanations for every decision.

### Core Gaps in Existing Systems

- Rule-based engines cannot adapt to new fraud patterns without manual updates
- Isolated transaction scoring ignores the relational context of entities
- Black-box ML models produce scores without explanations, making analyst review difficult
- Static risk thresholds do not respond to changing fraud velocity across the network
- Bias and fairness in automated blocking decisions is rarely audited

## 5. Core Objectives

The system must fulfill all five objectives stated in the hackathon problem statement:

**Objective 1 — Graph-Based Relational Modeling**
Model every financial entity (users, merchants, devices, IP addresses, bank accounts) as a node in a heterogeneous graph. Model every interaction (transactions, shared device usage, same-IP sessions) as typed edges. Use this graph as the primary data structure for fraud reasoning.

**Objective 2 — Anomaly Detection**
Detect three classes of anomalies: structural anomalies in the graph topology (new suspicious clusters forming), temporal anomalies in transaction sequences (velocity spikes, off-hours activity), and node-level anomalies scored by a Graph Neural Network that aggregates risk signals across the neighborhood of each entity.

**Objective 3 — Adaptive Thresholds**
Dynamically adjust fraud detection sensitivity based on real-time context — time of day, merchant category risk, geographic zone, and current network-wide fraud velocity. The system becomes more aggressive during fraud spikes and more conservative during low-risk periods to minimize false positives.

**Objective 4 — Explainability**
Every flagged transaction must come with a human-readable explanation generated by Claude Opus 4.5, identifying the fraud pattern, the key risk signals, a recommended action, and a confidence level — surfaced directly in the analyst dashboard.

**Objective 5 — Fairness and Bias Mitigation**
Continuously audit false positive rates across user segments. Flag and escalate decisions that appear to be influenced by non-fraud-relevant features. Maintain a visible fairness dashboard so the system can prove its own equity.

## 6. System Architecture Overview

FraudMesh is composed of five interconnected layers:

### Layer 1 — Data Ingestion
Transactions stream into the system in real time. Each transaction carries entity identifiers (user ID, merchant ID, device fingerprint, IP address, card number) and transaction metadata (amount, timestamp, channel, currency). In the hackathon prototype, a realistic transaction simulator generates this stream continuously, including hidden fraud ring patterns to test detection.

### Layer 2 — Graph Engine
As each transaction arrives, the graph engine updates a live heterogeneous property graph. Nodes represent entities: users, merchants, devices, IP addresses, and beneficiary accounts. Edges represent relationships: a user transacting with a merchant, a user accessing via a device, multiple users sharing the same device or IP. Shared device and shared IP edges are the primary structural fraud signals — they reveal when accounts are being operated by the same actor. The graph is queryable in real time for neighborhood traversal and feature extraction.

### Layer 3 — Detection Engine
The detection engine runs a three-layer analysis on every incoming transaction:

**Structural Analysis** uses community detection algorithms to identify when new suspicious clusters are forming in the graph — a hallmark of synthetic identity rings. It also monitors for sudden changes in graph topology that indicate coordinated account creation.

**Temporal Analysis** examines the sequence of transactions for each entity — detecting velocity spikes (many transactions in a short window), unusual timing patterns (activity at atypical hours), and abnormal geographic jumps between transactions.

**GNN Scoring** runs a Graph Neural Network that propagates risk signals across the entity graph through message passing. Each node aggregates risk information from its neighbors, meaning that a user connected to previously flagged entities receives elevated risk scores even if their own transactions look normal. The GNN outputs a fraud probability score between 0 and 1 for each transaction.

The final fraud score is an ensemble of all three layers, weighted by a meta-classifier trained on labeled fraud patterns.

### Layer 4 — Adaptive Threshold Engine
The threshold engine determines whether a given fraud score results in a block, a flag for review, or a pass. Crucially, the threshold itself is not static — it shifts based on:

- **Time context:** Lower thresholds (higher sensitivity) during late-night hours when fraud is more common
- **Amount context:** Lower thresholds for high-value transactions
- **Merchant risk profile:** Lower thresholds for high-risk merchant categories
- **Network fraud velocity:** When the system detects an elevated fraud rate across the network, it increases sensitivity globally until the spike subsides
- **False positive feedback:** Human analyst overrides feed back into threshold calibration

### Layer 5 — Explainability and Case Management (Claude Opus 4.5)
When a transaction is flagged, Claude Opus 4.5 receives the full fraud context — the GNN score, the triggered rules, the graph features, the transaction metadata, and the entity's behavioral history. Claude synthesizes this into a structured fraud report containing a headline summary, a narrative explanation connecting all the signals, the identified fraud pattern type, a recommended action, and a confidence rating. This report is displayed alongside the flagged transaction in the analyst dashboard, enabling one-click review decisions.

The Antigravity agentic layer extends this further — autonomously pulling the full transaction history of all connected graph nodes, cross-referencing patterns, building a complete fraud case file, and populating a case management queue without analyst initiation.

## 9. Data Model

### Entities (Graph Nodes)

| Entity Type | Key Attributes |
|-------------|---------------|
| User | user_id, account_age, risk_score, transaction_count |
| Merchant | merchant_id, category_code, risk_tier |
| Device | device_id, fingerprint, user_count (shared devices = fraud signal) |
| IP Address | ip_address, geo_location, user_count |
| Beneficiary Account | account_id, creation_date, linked_users |

### Relationships (Graph Edges)

| Edge Type | Connects | Risk Weight |
|-----------|----------|-------------|
| TRANSACTED_WITH | User → Merchant | Low (base) |
| USES_DEVICE | User → Device | Medium (high if shared) |
| SAME_IP_SESSION | User → IP | Medium (high if shared) |
| SHARES_DEVICE | User ↔ User | High — primary fraud signal |
| TRANSFERRED_TO | User → Beneficiary Account | High for new accounts |

### Transaction Record

Every transaction carries: transaction ID, user ID, merchant ID, device ID, IP address, amount, currency, timestamp, channel (web/mobile/POS), and a ground-truth fraud label (hidden from the model, used for evaluation).

## 10. Fraud Pattern Taxonomy

FraudMesh is designed to detect and label the following fraud pattern types:

**Account Takeover (ATO):** Legitimate user's credentials compromised. Signals include device change, new IP, unusual velocity, and high-value transactions to new beneficiaries.

**Synthetic Identity Fraud:** Fabricated identities constructed from real and fake data. Signals include new accounts with sparse history, shared devices across multiple new accounts, and rapid high-value transactions.

**Money Mule Operation:** Accounts used to receive and forward stolen funds. Signals include incoming high-value transfers followed immediately by outbound transfers, shared IP with flagged senders.

**Coordinated Fraud Ring:** Multiple accounts operated by the same actor. Signals include shared device fingerprints, shared IP addresses, synchronized transaction timing, and dense graph clustering among flagged nodes.

**Card-Not-Present Fraud:** Stolen card used for remote transactions. Signals include mismatch between cardholder's usual geography and transaction location, new device, unusual merchant category.

## 11. Fairness and Bias Framework

FraudMesh treats fairness as a first-class requirement, not an afterthought.

**Demographic Parity Monitoring:** The system tracks false positive rates across proxy segments including geographic region, transaction size band, and account age. If any segment's false positive rate exceeds 2x the system baseline, an alert is raised on the fairness dashboard.

**Counterfactual Fairness:** For borderline transactions (fraud score between 0.5 and 0.75), the system tests whether changing non-fraud-relevant features — merchant category, transaction channel, time of day — would flip the decision. If yes, the transaction is escalated for human review rather than auto-blocked.

**Analyst Override Tracking:** Every analyst override of a system decision is logged. Patterns in overrides (e.g., a particular region generating frequent false positives) are surfaced back to the threshold engine for calibration.

**Fairness Report:** A real-time fairness dashboard shows false positive rate by region, flag rate by transaction amount band, and demographic parity score over time.

## 13. User Interface — Screen Descriptions

### Screen 1 — Main Dashboard
The primary view. Divided into three zones: the live entity graph (left, large), the fraud alert feed (right, scrollable), and a metrics bar at the top showing current threshold value, network fraud rate, and system sensitivity mode. Dark theme throughout.

### Screen 2 — Alert Detail / Claude Explanation
Triggered by clicking any alert in the feed. Expands to show the full Claude Opus 4.5 fraud analysis: headline, narrative explanation, fraud pattern label, recommended action, confidence level, and the single most important risk signal. Also shows the raw fraud score and which rules were triggered.

### Screen 3 — Graph Explorer
Clicking any node in the live graph opens a neighborhood panel showing all first and second-degree connections, edge types, and risk scores. Fraud ring clusters are highlighted with a red overlay.

### Screen 4 — Fairness Dashboard
A dedicated tab showing false positive rate charts by segment, demographic parity score over time, and a log of analyst overrides. Segments in violation of the 2x threshold rule are highlighted in amber.

### Screen 5 — Threshold History
A line chart showing how the adaptive threshold has moved over the session, annotated with the contextual events that caused shifts (fraud spike detected, business hours override, high-value transaction adjustment).

## 14. Hackathon Demo Script

**Minute 0–1 — Opening Hook**
Open the dashboard with the live graph running. Point to nodes lighting up as transactions arrive. Show a fraud cluster forming in real time as the ring accounts start transacting — watch them turn red.

**Minute 2–3 — Detection in Action**
A high-risk transaction fires. Walk through the fraud alert card. Show the fraud score, the triggered signals, and the adaptive threshold meter. Explain why the threshold is where it is right now.

**Minute 4–5 — Claude Explanation**
Click the alert. Show Claude Opus 4.5's full fraud analysis — the headline, the narrative, the pattern type, the recommended action. Emphasize: this is what a fraud analyst actually needs. Not a number. A story.

**Minute 6 — Fairness Dashboard**
Switch to the fairness tab. Show the false positive rates by region. Show the demographic parity score. Make the point: the system proves its own fairness continuously.

**Minute 7 — Antigravity Agent**
Trigger the autonomous investigation agent on a flagged cluster. Watch it autonomously pull transaction histories, identify the ring structure, and generate a complete fraud case file — no analyst input required.

**Closing Line:** *"FraudMesh sees what isolated transaction scoring is blind to — the network. And it explains every decision in plain English, so analysts can act in seconds, not hours."*
